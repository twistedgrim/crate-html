Gotchas

Things I'm going to tell you because I assume you wouldn't figure them out on your own.

1. The daemon isn't magic

The daemon — "crated", with a "d" at the end, which in Unix tradition means "daemon," which is, I should mention, a Greek word originally — has to actually be running for any of this to work.

if you close the terminal you ran crated in, it stops. That's not a bug. That's how processes work. You may be thinking, "I'd like it to keep running." We'll get there. There's a thing called launchd. Don't worry about it yet.

2. The site name is not the directory name

I've seen people — not you, but people — type:

crate push ./mysite

And then wonder why it doesn't work. The command needs the directory and the name. Two arguments. Like a sandwich: bread, filling, more bread. Except there's no second bread. So just bread and filling.

the reason these are separate is so you can push the same directory under different names. Or different directories under the same name (which replaces the old one). It's about flexibility. I value flexibility. We all should.

3. Names have rules

Your site name must match this regex:

^[a-z0-9][a-z0-9._-]{0,62}$
regex stands for "regular expression," which is — you know what, I'll save that one. Just know: lowercase letters and digits to start, then letters/digits/dot/hyphen/underscore, max 63 chars. No spaces. No capital letters. No emojis. Especially no emojis. I see what you're trying to do.

4. Symlinks are silently dropped

If your directory contains symbolic links — "symlinks," in the parlance of our times — the daemon will not include them in the deployed site. It just... skips them. Quietly. Like a polite waiter clearing a half-eaten dish.

this is for security reasons. A symlink could point anywhere — /etc/passwd, your SSH key, the secret folder where you keep your unfinished novel. We don't want any of that getting served as static HTML. So: dropped.

5. It's only on localhost

The daemon binds to 127.0.0.1:7777, which means: only you, on this computer, can reach it. Nobody on your network. Nobody on the internet. Just you. That's by design. Don't try to "fix" it by making it bind to 0.0.0.0, because:

  1. There's no HTTPS yet.
  2. The token is in plaintext in config.yaml.
  3. You have not earned the right to bind to 0.0.0.0.
if you genuinely need remote access, the answer is Caddy in front of the daemon. Or Tailscale. There's a roadmap. It's right here. I've thought about this more than you have.

6. The token is in your config file

On first run, the daemon generates a random bearer token and stores it in config.yaml. If you delete that file, you'll get a new token, which means any other client (like, I don't know, a script you wrote) using the old token will stop working. So don't do that.

the file has mode 0600, which means only you can read it. Which is correct. Which is good. Which I made sure of, because I think about these things.
